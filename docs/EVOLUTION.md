# 프로젝트 진화 과정 (Evolution)

> 이 프로젝트는 단발성 구현이 아니라 **최초 PoC 명세 → 분야 전문가 자문 → 반복 개선**의 사이클로 발전했습니다.
> 아래는 실제 개발 과정에서 무엇을, 왜 바꿨는지에 대한 기록입니다.

---

## 1. v1 — 최초 PoC (baseline)

**목표**: 비전문가가 데이터와 분석 목표를 입력하면 LLM 멀티에이전트가 ML 모델을 자동 생성·실행해 결과를 보여주는 시스템의 기술적 실현 가능성 검증.

**당시 범위와 한계 (의도된 단순함)**

| 영역 | v1 상태 |
|------|---------|
| 방법론 검증 | **정규식 기반** 게이트 3종 (데이터 누수 · SMOTE 순서 · 평가 무결성) |
| 데이터 형식 | CSV 단일 |
| 분석 유형 | 분류 / 회귀 |
| 세션 저장 | 인메모리 (프로세스 재시작 시 소실) |
| 개인정보 | 별도 처리 없음 |
| 개선 방식 | 코드 생성 실패 시 재시도 루프 1개 |
| 진입 모드 | Mode A(자연어) + Mode B(명세서) 병존 |

> 설계 원칙은 처음부터 명확히 잡았습니다 — 3-tier 레이어 분리, `LLMProvider` 추상화, AutoML 라이브러리 대신 LLM이 직접 코드를 생성하는 투명한 구조.

---

## 2. 분야 전문가 자문

사전 공유한 아키텍처 문서를 바탕으로 **AI 공학 전공 교수의 자문**을 받았습니다. 핵심 지적 사항:

**총괄 방향**: "처음부터 다 넣지 말고, 심플하게 완성한 뒤 점진적으로 추가하라."

| 지적 사항 | 내용 |
|-----------|------|
| 🔴 방법론 검증의 근본 약점 | 정규식 검출은 **동적 변수명으로 우회 가능** → AST(추상 구문 트리) 기반으로 전환 필요 |
| 검증 항목 부족 | 교차검증 무결성 · 다중공선성 · 재현성 검사 추가 |
| 실행 격리 없음 | `exec()` 기반 → **Docker 컨테이너 격리** 권고 |
| 세션 영속성 | 인메모리 → Redis / PostgreSQL |
| 동시성 | 단일 프로세스 → Celery + 워커 풀 |
| 개인정보 | 익명화 로직을 **Data Layer**에 배치 |
| 데이터 확장성 | CSV 외 다양한 형식 + Parquet / DB 연동 |
| 모드 구조 | Mode B 중심으로 단일화 + 입력 템플릿 제공 |
| 개선 루프 | 전처리 루프 · 성능 개선 루프 추가 |

## 3. 반영한 개선 (v1 → 현재)

자문 내용 중 **우선순위가 높고 코드 레벨에서 바로 반영 가능한 것**부터 구현했습니다.

| 개선 | Before | After | 관련 파일 |
|------|--------|-------|-----------|
| **방법론 검증 재작성** | 정규식 3종 (문자열 매칭, 우회 가능) | **AST 기반 6종** — 변수 별칭 체인 추적으로 이름 변경도 탐지 | `agents/ml_gates.py` |
| **검증 항목 확장** | 3종 | +교차검증 무결성 · 시계열 분할 · 재현성 고정 | `agents/ml_gates.py` |
| **결과 진단 강화** | 기본 성능 지표 | +다중공선성 · 피처 쏠림(누수 시그널) 룰 체크 | `agents/eval_agent.py` |
| **개선 루프 추가** | 코드 생성 재시도 1개 | **이중 루프** — 안쪽(게이트 위반 재생성) + 바깥(성능 미달 시 재시도, 최대 3라운드) | `agents/orchestrator.py` |
| **개인정보 마스킹** | 없음 | Data Layer에 PII 탐지·마스킹 추가, LLM 호출 지점에서만 마스킹 | `loaders/anonymizer.py` |
| **데이터 형식 확장** | CSV | CSV·TXT·TSV·Excel·JSON + 폴더 단위/다중 파일 병합(행·키·시간 기반) | `loaders/csv_loader.py` |
| **시계열 지원** | 분류/회귀만 | lag-feature 기반 시계열 task_type 추가 | `agents/plan_agent.py` |
| **명세서 자동 분석** | 수동 입력 | 명세서 텍스트에서 컬럼 설명·타겟·유형 자동 추출 | `agents/spec_agent.py` |
| **비용 추적 + 캐싱** | 없음 | 토큰/비용 추적 + 동일 요청 캐시(재현성·비용 절감) | `providers/caching_provider.py` |
| **LLM 출력 파싱 견고화** | 코드 앞에 설명 문장이 붙으면 문법 오류로 실패 | 펜스 블록을 위치와 무관하게 추출, 회귀 테스트로 방어 | `agents/code_agent.py` |
| **모드 단일화** | Mode A/B 병존 | Mode B 중심으로 단일화 (Mode A 코드는 보존) | `ui/mode_b.py` |

## 4. 향후 로드맵 (자문에서 나온 미반영 항목)

"먼저 심플하게 완성" 원칙에 따라 아래는 다음 단계로 계획하고 있습니다. (상세: [README의 제약 및 향후 로드맵](../README.md#제약-및-향후-로드맵))

- **실행 격리**: `exec()` → Docker 컨테이너 샌드박스
- **세션 영속성**: 인메모리 → Redis / PostgreSQL
- **동시성**: 단일 프로세스 → Celery + 워커 풀
- **재현성 관리**: MLflow 도입
- **데이터 확장**: Parquet · DB 직접 연결 · API 연동
- **입력 템플릿**: Mode B에 설문형 입력 템플릿 제공

> 각 개선이 **기존 코드 변경 없이 구현체 교체로 가능하도록** `LLMProvider`·`SessionStore`를 추상 인터페이스로 미리 설계해 두었습니다.
