# 다이어그램 설명 가이드 — 남에게 설명하기 위한 문서

> 이 문서는 [`architecture-diagrams.md`](./architecture-diagrams.md)의 9개 다이어그램을 **비전문가도 이해하고, 직접 남에게 설명할 수 있도록** 풀어 쓴 가이드다.
> 각 다이어그램마다 `무엇을 보여주나 → 어떻게 읽나 → 이렇게 설명하세요` 순서로 정리했다.

---

## 0. 시작 전에 — 30초 엘리베이터 피치

먼저 다이어그램 없이, 시스템 전체를 한 문장으로 말할 수 있어야 한다.

> **"데이터 파일만 올리면, AI가 알아서 분석 계획을 세우고 → 머신러닝 코드를 직접 짜고 → 모델을 학습시키고 → 결과가 믿을 만한지까지 진단해주는 자동화 도구입니다. 사람이 코딩이나 통계를 몰라도 됩니다."**

이게 핵심이다. 나머지 다이어그램은 전부 "그래서 그게 내부에서 어떻게 돌아가는데?"에 대한 답이다.

---

## 0-1. 먼저 알아야 할 5가지 용어 (비유로)

설명하다 보면 반드시 나오는 단어들이다. 비유를 통째로 외워두면 편하다.

| 용어 | 쉬운 뜻 | 비유 |
|------|--------|------|
| **3-Tier(3계층)** | 프로그램을 역할별로 3층으로 나눈 구조 | **식당**: 홀 직원(주문받기) / 주방(요리) / 창고(재료보관) |
| **에이전트(Agent)** | 한 가지 일만 전담하는 AI 일꾼 | 분야별 **전문가** (기획자, 코더, 검수자) |
| **오케스트레이터(Orchestrator)** | 에이전트들을 순서대로 부리는 총괄 | **지휘자** 또는 프로젝트 매니저(PM) |
| **LLM** | 글을 이해하고 생성하는 AI (Claude) | 똑똑한 **신입 박사** — 시키면 코드도 쓰고 글도 씀 |
| **게이트(Gate)** | 통과 기준을 검사하는 관문 | 공장의 **품질검사관** |

> 💡 **설명 팁**: "이 시스템은 AI 한 명이 다 하는 게 아니라, **역할이 다른 AI 여러 명이 팀**으로 일하고, 그 위에 **PM 격인 오케스트레이터**가 순서를 지휘하는 구조예요."

## 1. 시스템 컨텍스트 — "큰 그림"

**무엇을 보여주나**
시스템을 멀리서 본 조감도. 우리 앱이 누구와(사용자), 무엇과(Claude AI, 데이터 파일) 연결되는지만 보여준다.

**어떻게 읽나**
- 가운데 큰 박스가 **우리 앱**. 그 안은 3층(Presentation / Logic / Data)으로 나뉘어 있다.
- 왼쪽 사람이 **사용자**, 오른쪽 구름이 **Claude AI(외부)**, 아래가 **데이터 파일**.
- 화살표는 "누가 누구에게 무엇을 주고받나"를 뜻한다.

**이렇게 설명하세요**
> "사용자는 앱에 데이터랑 목표만 줍니다. 앱은 머리를 굴려야 할 때 외부의 Claude AI에게 물어보고, 데이터는 파일에서 읽어옵니다. 우리 앱이 가운데서 이 모든 걸 중개하는 거예요."

**예상 질문 대비**
- *"Claude를 왜 밖에 두나요?"* → "Claude는 우리가 만든 게 아니라 빌려 쓰는 외부 AI 서비스라서요. 나중에 보안 때문에 회사 내부 AI로 바꿀 수도 있는데, 그 구조는 9번 다이어그램에서 설명드릴게요."

---

## 2. 3-Tier 컴포넌트 구조 — "건물의 층별 안내도"

**무엇을 보여주나**
앱 내부를 3개 층 + 1개 보조(도메인)로 나눠, 각 층에 어떤 파일(부품)이 있는지 보여준다.

**어떻게 읽나**
- **위층 Presentation(화면)**: 사용자가 보는 화면. 입력 받고 결과 보여주는 일만 함.
- **가운데 Logic(두뇌)**: 실제 판단과 일처리. 에이전트들이 여기 산다.
- **아래층 Data(창고/통로)**: 파일 읽기, 코드 실행, AI 호출 등 바깥과 소통.
- 화살표는 **위에서 아래로만** 흐른다 (위층이 아래층을 부림, 그 반대는 없음).

**이렇게 설명하세요**
> "식당으로 비유하면, 화면은 홀 직원, 로직은 주방, 데이터는 창고예요. 홀 직원은 손님 응대만 하고 요리는 주방에 맡기죠. 주방은 재료를 창고에서 가져오고요. **각자 자기 일만 하고, 순서를 건너뛰지 않는다** — 이게 이 구조의 핵심 규칙입니다."

**왜 이렇게 나누나 (중요)**
> "이렇게 층을 나누면, 나중에 화면 디자인을 바꿔도 주방(로직)은 안 건드려도 되고, AI를 Claude에서 다른 걸로 바꿔도 윗층은 그대로예요. **고치기 쉽고 안 망가지는** 구조를 위한 겁니다."

## 3. Mode A 시퀀스 — "자동 모드, 시간 순서대로"

**무엇을 보여주나**
사용자가 "분석 시작"을 누른 뒤, 결과가 나올 때까지 내부에서 일어나는 일을 **위에서 아래로 시간 순서**대로 나열한 것.

**어떻게 읽나**
- 맨 위 가로줄들이 **등장인물**(사용자, UI, 오케스트레이터, 각 에이전트, Claude).
- 세로로 내려가며 화살표가 **대화(호출)** 를 나타낸다. 위가 먼저, 아래가 나중.
- 숫자(①②③…)가 일의 순서다.

**핵심 6단계 (이 순서만 기억하면 됨)**
1. **데이터 로드** — 파일을 읽고 구조 파악
2. **계획(PlanAgent)** — "어떤 알고리즘으로 풀지" Claude가 결정
3. **코드 생성(CodeAgent)** — Claude가 실제 ML 코드를 작성
4. **검증(Validator + 게이트)** — 그 코드가 안전하고 올바른지 검사
5. **실행(Executor)** — 코드를 돌려 모델을 학습
6. **진단(EvalAgent)** — 결과가 믿을 만한지 평가

**이렇게 설명하세요**
> "버튼 하나 누르면 안에서 이 6단계가 자동으로 쭉 돌아갑니다. 데이터 읽고 → AI가 계획 세우고 → AI가 코드 짜고 → 그 코드를 검사하고 → 실행해서 모델 만들고 → 결과를 진단하는 거죠. 사람은 처음에 목표만 주고, 마지막에 결과만 받습니다."

> 💡 **설명 팁**: 화살표 하나하나 짚지 말고, **6단계 덩어리**로 묶어서 말하라. 디테일을 물으면 그때 짚으면 된다.

---

## 4. Mode B 시퀀스 — "추천받고 고르는 모드"

**무엇을 보여주나**
Mode A와 거의 같지만, **중간에 사람이 한 번 개입**하는 점이 다르다. AI가 분석 방향 3가지를 제안하면, 사람이 하나를 고른 뒤 진행한다.

**어떻게 읽나**
- 위쪽 연한 보라 영역 = **1단계: AI가 방향 3개 제안**
- 아래쪽 연한 하늘색 영역 = **2단계: 사람이 고른 방향으로 실행**
- 1단계가 끝나면 일단 **멈추고 사람의 선택을 기다린다**.

**Mode A와의 차이 (이것만 말하면 됨)**
> "Mode A는 AI가 알아서 다 하는 완전 자동이고, Mode B는 **AI가 메뉴 3개를 추천해주면 사람이 골라서** 진행하는 반자동이에요. 데이터를 잘 아는 사람이 '나는 이 방향이 좋겠다'고 통제하고 싶을 때 쓰는 모드입니다."

**예상 질문 대비**
- *"왜 멈췄다 다시 가나요?"* → "사람의 선택을 받아야 하니까요. 그래서 앱이 '선택 대기' 상태로 잠깐 멈춰 있다가, 고르면 나머지(코드 생성→실행→진단)를 이어서 합니다. 그 뒤 절반은 Mode A랑 완전히 똑같아요."

---

## 5. 세션 상태 머신 — "작업의 신호등"

**무엇을 보여주나**
하나의 분석 작업이 시작부터 끝까지 거치는 **상태**들과, 어떤 조건에서 다음 상태로 넘어가는지.

**어떻게 읽나**
- 각 동그라미/박스가 **상태**(대기 → 계획중 → 코드생성중 → 실행중 → 완료).
- 화살표 위 글씨가 **넘어가는 조건**.
- 어느 단계든 문제가 생기면 **FAILED(실패)** 로 빠진다.
- Mode B만 중간에 **"선택 대기"** 상태가 추가된다.

**이렇게 설명하세요**
> "택배 배송 조회 같은 거예요. '접수 → 상품준비 → 배송중 → 배송완료'처럼, 분석 작업도 '대기 → 계획 → 코드생성 → 실행 → 완료' 단계를 거칩니다. 지금 어느 단계인지 시스템이 항상 알고 있어서, 화면의 진행 표시도 이걸 보고 보여주는 거예요."

> 💡 **설명 팁**: 화면에서 본 **진행 파이프라인 표시(칩들)** 가 바로 이 상태를 시각화한 거라고 연결하면 이해가 빨라진다.

## 6. 코드 생성 재시도 루프 — "AI가 짠 코드를 검사하고 되돌려보내기"

**무엇을 보여주나**
CodeAgent가 코드를 짤 때, 한 번에 끝내는 게 아니라 **검사 → 불합격이면 다시 짜기**를 반복하는 과정.

**어떻게 읽나**
- AI가 코드를 짠다 → **게이트(검사관)** 가 방법론 오류를 확인한다.
- 통과하면 끝. 통과 못 하면 **"이런 게 틀렸다"고 알려주며 다시 짜게** 한다.
- 단, 무한 반복은 막으려고 **최대 3번**까지만 시도한다.

**무엇을 검사하나 (쉽게)**
- **데이터 누수**: 시험 문제를 미리 보고 공부하는 것 같은 부정행위 (정답을 몰래 참고)
- **평가 방법 오류**: 연습문제로만 채점하고 진짜 시험은 안 본 것
- **순서 오류**: 데이터를 부풀리는 처리를 잘못된 시점에 한 것

**이렇게 설명하세요**
> "AI가 코드를 짜긴 하는데, AI도 실수를 하잖아요. 그래서 **자동 검사관**을 둬서, ML에서 흔히 저지르는 반칙(예: 시험 문제를 미리 보는 식의 '데이터 누수')을 잡아냅니다. 걸리면 '이거 틀렸으니 다시 짜'라고 AI한테 돌려보내요. 최대 3번까지 고쳐 짭니다. **AI를 그냥 믿는 게 아니라 검증하는** 안전장치죠."

> 💡 이 부분이 이 제품의 **차별점**이다. "AI가 코드 짜주는 건 많지만, 그 코드가 통계적으로 올바른지까지 자동 검증하는 건 흔치 않다"고 강조하면 좋다.

## 7. 결과 진단 로직 — "이 결과 믿어도 돼?"

**무엇을 보여주나**
모델이 나온 뒤, 그 결과가 신뢰할 만한지 판정하는 과정. **기계적 규칙 검사 + AI의 말풀이**가 합쳐진다.

**어떻게 읽나**
- 왼쪽: 정해진 **규칙으로 체크**(샘플 수가 충분한가, 성능이 너무 낮거나 의심스럽게 높지 않은가, 데이터가 한쪽으로 쏠리지 않았나).
- 가운데: 체크 결과를 종합해 **신호등 판정**.
  - ✅ 신뢰 가능 / ⚠️ 주의 필요 / ❌ 신뢰 어려움
- 오른쪽: 그 판정을 **AI가 비전문가용 설명**으로 풀어줌 (요약·리스크·개선안).

**이렇게 설명하세요**
> "보통 AI 도구는 '정확도 94%' 숫자만 던져주고 끝나요. 근데 그 숫자가 진짜 믿을 만한 건지는 또 다른 문제거든요. 이 시스템은 '샘플이 너무 적진 않나', '수상하게 정확도가 높진 않나(부정행위 의심)' 같은 걸 **자동으로 점검**해서 신뢰 가능/주의/위험 신호등을 켜주고, 왜 그런지 **사람 말로 설명**까지 해줍니다."

> 💡 6번(코드 검증)과 7번(결과 진단)이 **'AI를 못 믿으니 이중으로 검증한다'**는 같은 철학의 두 안전장치임을 묶어서 말하면 설득력이 커진다.

## 8. 도메인 모델 관계 — "데이터가 담기는 상자들"

**무엇을 보여주나**
시스템 안에서 정보가 어떤 **상자(데이터 구조)** 에 담겨 오가는지, 상자끼리의 포함 관계.

**어떻게 읽나**
- 각 박스가 **데이터 묶음**(세션, 스키마, 계획, 결과 등).
- 가장 큰 **세션(Session)** 상자 안에 스키마·계획·결과가 들어 있다.
- 선은 "A가 B를 가지고 있다" 같은 **포함/연결 관계**.

**이렇게 설명하세요**
> "이건 좀 더 기술적인 그림인데, 한마디로 **'정보를 어떤 서류 봉투에 담아 주고받느냐'** 예요. '세션'이라는 큰 봉투 하나에 그 작업의 데이터 구조, 분석 계획, 최종 결과가 다 들어갑니다. 작업 하나당 봉투 하나라고 보면 돼요."

> 💡 **설명 팁**: 비개발자 청중이면 이 장은 **가볍게 넘기거나 생략**해도 된다. "내부적으로 데이터를 정리해두는 방식" 정도로만 언급. 개발자 청중이면 자세히.

## 9. LLM 교체 지점 — "AI를 갈아끼울 수 있는 콘센트"

**무엇을 보여주나**
에이전트들이 Claude AI에 직접 묶여 있지 않고, **표준 규격(인터페이스)** 을 통해 연결돼 있어서, AI를 다른 걸로 쉽게 바꿀 수 있다는 구조.

**어떻게 읽나**
- 위: 에이전트 4명. 이들은 **표준 콘센트(LLMProvider)** 에만 연결된다.
- 아래: 그 콘센트에 **Claude(클라우드)** 또는 **로컬 AI(회사 내부 서버)** 를 꽂을 수 있다.
- 초록색이 새로 추가 가능한 옵션(로컬 모델).

**이렇게 설명하세요 (영업·보안 포인트)**
> "지금은 Claude를 쓰지만, **콘센트 규격만 맞추면 다른 AI로 갈아끼울 수 있게** 설계했어요. 예를 들어 보안이 엄격해서 외부 인터넷을 못 쓰는 회사라면, Claude 대신 **회사 서버 안에서 도는 AI**를 꽂으면 됩니다. 핵심 로직(에이전트들)은 하나도 안 바꿔도 돼요. 데이터가 외부로 안 나가니 보안 고객도 안심할 수 있죠."

> 💡 이 장은 **판매·도입 협상에서 가장 강력한 카드**다. "당신 회사 보안 정책에 맞출 수 있다"는 메시지이기 때문.
